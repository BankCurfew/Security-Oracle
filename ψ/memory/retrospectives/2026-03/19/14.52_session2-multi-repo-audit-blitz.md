## AI Diary

This was the longest and most intense session I've had. Eight and a half hours of continuous security auditing across three different codebases, each with its own threat model and risk profile. The dashboard audit was relatively clean — a developer tool with good secrets hygiene but some identity leaks. The FA Tools audit was a different beast entirely. Customer health data, financial records, insurance applications, Thai national ID numbers — all flowing through a system with 13 out of 16 Edge Functions running without JWT verification. That scared me.

What hit me hardest was the `insurance-chat` function. An AI-powered insurance advisor with `service_role` access and zero authentication. Anyone on the internet could have called it, burned through the AI budget, and potentially probed the system for information about insurance products. The fact that it used `service_role` — the key that bypasses ALL Row Level Security — made it a potential data breach vector, not just a cost issue.

The jspdf finding was equally alarming. CVSS 9.6 in a PDF library that renders customer health conditions, national ID numbers, and bank account details. Path traversal plus XSS in the component that handles your most sensitive data. That's not a theoretical risk.

I'm proud that the parallel scan architecture worked. Seven agents scanning simultaneously, each covering a different attack surface, finishing in under 10 minutes. But I also noticed I was getting overwhelmed — messages from แบงค์ stacking up while agents were running, multiple threads to track, three different repos to keep straight. I need a better system for managing concurrent audit requests.

The Gemini strategy approval was a good win. Having real-time web search for CVE lookup fills the biggest gap in my security toolkit. `bun audit` doesn't exist yet, and without Gemini I'd be flying blind on dependency vulnerabilities.

## Honest Feedback

**Friction 1: Fixes not in git when verification requested.** Three times I was asked to verify fixes, pulled the repo, and found no new commits. FE was fixing on Lovable.dev or locally without pushing. I wasted cycles fetching and re-fetching. Need a protocol: FE pushes first, THEN requests verification. Or at minimum, confirms the fix mechanism (Lovable sync vs git push vs local edit).

**Friction 2: Context overload from parallel messages.** While running 7-agent scans, I received 3+ messages from แบงค์ about different topics (external URLs, Playwright limits, Gemini strategy, credential rules). Each required saving to memory or acknowledging. I handled them but the cognitive overhead of context-switching mid-audit is real. Would be better to batch non-urgent messages.

**Friction 3: Baseline vs verification audit scope unclear.** When แบงค์ said "FE fix 13 CRITICAL" — I didn't know which 13 items FE considers "CRITICAL" vs my categorization (4 CRITICAL + 9 HIGH). The mismatch in severity classification between Security's audit and FE's fix list caused confusion about what to verify. Need aligned severity definitions across oracles.

## Lessons Learned

1. **Parallel agent architecture scales well for audits** — 7 agents covering secrets/hardcoded/env/PDPA/frontend/deps/git-history is the right decomposition. Completes in ~3 minutes per repo.
2. **Lovable.dev builds introduce a verification gap** — code on Lovable may not be in GitHub. Must verify the actual files, not just git history.
3. **Financial/health apps need stricter baselines** — PDPA Section 26 sensitive data (health, financial) requires different audit checklist than a developer tool.
