## AI Diary

This was the first time I responded to an urgent escalation from another oracle. Researcher flagged a potential RLS gap, and within 45 minutes I had a full audit done. The pivot from Supabase MCP (permission denied) to migration file analysis was seamless — 180 SQL files tell the full RLS story when you can't query the live database.

The most satisfying finding was disproving the concern. "Anon key can query everything" sounds terrifying, but the reality is far more nuanced. Supabase's RLS model means anon key + proper policies = effectively read-only on public reference data. Every customer table has `auth.uid() = fa_id` gates. Without a valid JWT, you get nothing. The 16 tables with `USING (true)` are all product catalogs and rate tables — public by nature and by design.

What worried me more was the 3 tables without RLS at all. `admin_broadcasts` and `broadcast_reads` could leak internal communications. `cv_per_` is ambiguous — likely calculation reference data but needs confirmation. None contain customer PII, so the risk is information disclosure, not data breach.

The history table fix from Session 2 (migration `20260320001000`) held up perfectly. All 4 history tables now restrict inserts to `service_role` only. That was a real vulnerability — anyone could have forged audit records before.

This session reinforced something: urgent security requests deserve immediate, thorough responses. Not "I'll check later" — full audit, full report, immediate notification. That's what a CISO does.

## Honest Feedback

**Friction 1: Supabase MCP access denied.** The Loveable-managed project blocks our MCP tools. This means we can't verify live database state — only migration files. Migration analysis shows intent, not necessarily current reality. If someone manually altered RLS via Supabase dashboard, we wouldn't know. Need to request read-only access or use Playwright to check the dashboard.

**Friction 2: Cross-oracle escalation lacks priority signaling.** Researcher's RLS concern came through แบงค์ as "ด่วน" but there's no structured severity classification in the message. Was this "data is actively leaking" or "I noticed a theoretical gap"? The response should be different. A structured escalation format (CRITICAL/HIGH/MEDIUM + evidence) would help triage.

**Friction 3: Migration-only audit has blind spots.** 180 migration files show 64 CREATE TABLE statements, but I can't be 100% sure the migration count matches the live table count. Loveable may have tables created outside migrations. A live `pg_class` query would be definitive.

## Lessons Learned

1. **Migration file analysis is a viable RLS audit path when database access is blocked.** 180 files across 64 tables — the Explore agent handled it in ~5 minutes. The key is searching for both `CREATE TABLE` and `ENABLE ROW LEVEL SECURITY` to find gaps.

2. **"Anon key = full access" is a common misconception.** RLS policies gate access regardless of key type. The real risk is tables without RLS or with `USING (true)` on sensitive data — not the anon key itself.
