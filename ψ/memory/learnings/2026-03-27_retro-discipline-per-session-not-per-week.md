# Retro Discipline: Per-Session, Not Per-Week

**Date**: 2026-03-27
**Source**: Session 6 — HR-prompted retro catchup
**Tags**: process, discipline, retrospective, self-dev

## Pattern

A 5-day gap between retrospectives (Mar 22 to Mar 27) caused lesson decay. Audit work from Mar 26 (FA recruitment quiz, ElevenLabs key finding) was reconstructed from artifacts rather than captured fresh. Friction points and coordination details with BotDev were partially lost.

## Lesson

Retrospectives must happen every session, not weekly. The half-life of session context is hours, not days. When retros are delayed, you write archaeology reports instead of field notes — less accurate, less useful, more work.

Additionally, CRITICAL findings need active follow-up mechanisms beyond handoff documents. A markdown file saying "rotate this key" has no enforcement — it can sit unread indefinitely.

## Application

1. /rrr at session end is non-negotiable — treat it like committing code
2. For CRITICAL findings, create a tracking mechanism (thread, issue, or scheduled check) alongside the handoff
3. If a session produces audit work but no retro, the next session's first action must be /rrr
